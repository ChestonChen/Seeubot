Interview Block

1. 项目定位与产品价值

39 questions
Q001

这个项目一句话怎么介绍？

参考回答：Seeubot 是一个本地优先的 AI Coding Agent 监控组件，用胶囊/灵动岛形式实时展示 Claude Code、Codex、Cursor 的会话状态、working/idle 数量和 token 使用情况。它的价值不是替代 agent，而是让用户在桌面层面获得低打扰的运行态感知。

Q002

为什么说这个项目有实际价值？

参考回答：AI agent 任务经常在后台运行，用户很难知道是否仍在工作、是否卡住、哪类工具正在消耗 token。Seeubot 把这些状态集中到桌面组件里，降低用户切换窗口和查看日志的成本。

Q003

项目解决的核心痛点是什么？

参考回答：核心痛点是多 agent 并行使用时缺乏统一、实时、低打扰的状态反馈。尤其是 Cursor、Claude Code、Codex 的数据源和进程形态不同，用户需要一个统一抽象来理解它们是否正在工作。

Q004

为什么选择做桌面组件而不是 Web Dashboard？

参考回答：agent 使用场景发生在本地 IDE 和终端中，桌面组件能常驻最前、实时反馈、无需打开浏览器。Web Dashboard 更适合历史分析，但不适合低打扰的即时状态感知。

Q005

这个项目和普通菜单栏工具有什么区别？

参考回答：普通菜单栏工具更多是入口或静态状态，Seeubot 强调动态胶囊、展开看板和工作态动画。它把实时监控信息压缩到屏幕顶部，类似系统级状态组件。

Q006

为什么要支持 Claude Code、Codex、Cursor 三类 agent？

参考回答：这三类工具覆盖了终端 agent、GUI agent、IDE 内 agent 的典型形态。支持它们可以验证采集抽象是否足够通用，也能体现跨工具统一监控的价值。

Q007

为什么 Cursor 支持是项目亮点？

参考回答：Cursor 没有像 Claude/Codex 那样稳定暴露 token 字段和终端会话形态，需要结合 agent-transcripts、usage cache 和进程信息进行判断。这体现了对复杂本地数据源的适配能力。

Q008

这个项目适合写在简历里吗？

参考回答：适合，原因是它不仅是 UI 小组件，还包含跨平台桌面开发、本地数据采集、状态建模、安装分发、隐私设计和动画体验优化。简历表达时应强调工程完整度和跨端落地。

Q009

项目的用户是谁？

参考回答：主要用户是高频使用 AI Coding Agent 的开发者，尤其是同时使用 Cursor、Claude Code、Codex 的用户。它帮助用户快速判断当前 agent 是否工作、是否空闲、token 消耗如何。

Q010

项目如何体现产品思维？

参考回答：它从“agent 在跑但用户不知道状态”这个具体痛点出发，设计了胶囊态和展开态两级信息架构。胶囊态只展示 working/idle，展开态再展示 token 和 session，符合低打扰原则。

Q011

为什么要有 bot mascot？

参考回答：bot mascot 提供情绪化状态反馈，让 working/idle/sleeping 不只是数字变化。它增强可感知性，也让项目更像完整产品而不是纯工具。

Q012

项目最重要的取舍是什么？

参考回答：最重要的取舍是本地优先与精确统计之间的平衡。为了隐私和易安装，项目优先读取本地文件和进程信息；对于无法精确统计的数据源，选择保守展示而不是伪造精度。

Q013

为什么不做服务端同步？

参考回答：服务端同步会引入隐私、登录、网络依赖和部署成本，而项目目标是轻量桌面监控。本地读取足以覆盖核心场景，且更符合开发者对代码和 token 数据的敏感性。

Q014

这个项目如何体现完整工程闭环？

参考回答：它包含项目官网、安装脚本、macOS app、Windows app、采集逻辑、UI 动画、自启动、卸载、GitHub Pages 和分发 zip，覆盖从开发到交付的完整链路。

Q015

如果面试官质疑项目只是 UI 包装，怎么回答？

参考回答：可以强调 UI 只是结果，难点在于不同 agent 数据源的抽象、实时状态判断、跨平台窗口行为、打包分发和稳定动画体验。尤其 Cursor 的采集与 Windows 端交互都不是简单 UI。

Q016

项目和原始 Seeybot 的关系是什么？

参考回答：最初基于 Seeybot 思路和仓库进行 fork 定制，后续改造成 Seeubot，扩展了 Cursor 支持、动态 agent 展示、Windows 独立实现、官网改造和安装分发。

Q017

为什么 fork 后还要改安装命令？

参考回答：安装脚本里的仓库地址决定 curl 下载和更新来源。fork 成自己的项目后必须将安装命令、GitHub 链接、更新检查和 README 都指向自己的仓库。

Q018

为什么项目命名从 Seeybot 到 Seeubot 有意义？

参考回答：命名变化代表项目所有权和品牌识别，但需要同步修改 app bundle、脚本、LaunchAgent、官网链接等多个位置，否则安装和运行会出现路径不一致。

Q019

这个项目的展示链接有什么作用？

参考回答：展示链接承担项目介绍、安装入口和简历作品展示作用。它让面试官不用读源码就能理解产品形态、功能范围和使用方式。

Q020

官网为什么要强调 Claude、Codex 和 Cursor？

参考回答：这是项目的核心能力边界。直接展示支持的 agent 能让用户快速判断是否适合自己的工作流，也能体现从两类工具扩展到多工具监控的价值。

Q021

为什么要去掉官网多余模块？

参考回答：项目官网的目标是说明价值和引导安装，过多语言 tab、更新日志、footer、访客统计会稀释重点。简化页面能提升表达效率。

Q022

项目有哪些可量化成果？

参考回答：可量化点包括支持 3 类 agent、macOS/Windows 双端、一行命令安装、1 秒级状态刷新、Cursor 10 分钟 idle 保留、400+ 面试问答沉淀等。

Q023

项目最能打动面试官的点是什么？

参考回答：不是某个单点技术，而是从真实使用痛点出发，持续迭代到跨平台、可安装、可展示、可维护的完整产品。多次 UI 细节优化也体现了产品打磨能力。

Q024

如果只讲三条核心功能，会怎么讲？

参考回答：第一，统一监控 Claude Code、Codex、Cursor 的本地会话状态。第二，展示 working/idle、token 与 live sessions。第三，提供 macOS 灵动岛和 Windows 胶囊组件的一行安装体验。

Q025

如果只讲两条项目亮点，会怎么讲？

参考回答：第一，本地优先采集多 agent 状态，兼顾隐私和实时性。第二，从 macOS 原生扩展到 Windows Tauri，保持跨端一致的交互体验。

Q026

项目为什么不是简单脚本？

参考回答：因为它涉及长期运行的桌面窗口、实时状态模型、跨平台 UI、安装自启动、文件解析、进程识别和视觉反馈。脚本只能输出一次结果，无法提供持续交互体验。

Q027

项目为什么适合 AI 工具生态？

参考回答：AI Coding Agent 正在从单次问答变成后台执行任务，状态监控会成为刚需。Seeubot 正好位于用户和多个 agent 之间，提供统一状态层。

Q028

项目会不会过度依赖特定工具内部实现？

参考回答：有一定依赖，尤其 Cursor transcript 和 usage cache 不是稳定公共 API。项目通过保守解析、容错和可扩展 adapter 结构降低这种风险。

Q029

如果 Cursor 改了本地路径怎么办？

参考回答：应把 Cursor 数据源封装在独立 adapter 中，并支持路径候选、缺失降级和日志调试。这样路径变化只影响单个 adapter，不影响整体架构。

Q030

这个项目未来可以怎么商业化或产品化？

参考回答：可以扩展为多 agent 控制台，支持任务历史、成本分析、异常提醒、团队汇总和更多 IDE/CLI agent。当前版本更像个人效率工具和技术展示。

Q031

为什么选择实时看板而不是通知提醒？

参考回答：通知适合异常或完成事件，但频繁通知会打扰。实时看板更适合持续状态感知，用户需要时看一眼即可。

Q032

项目中最容易被忽视的价值是什么？

参考回答：安装和卸载体验。很多个人项目只停留在源码能跑，Seeubot 提供一行安装、自启动和卸载命令，更接近真实用户可用的软件。

Q033

面试介绍时如何避免讲得太散？

参考回答：按“痛点-方案-难点-结果”讲：多 agent 状态不可见；做本地桌面组件；解决采集和跨端 UI；最终支持三类 agent 和双端安装。

Q034

如果面试官问为什么不用现成监控工具？

参考回答：现成监控工具通常关注 CPU、进程或日志，不理解 agent 会话、working/idle 和 token。Seeubot 是面向 AI Coding Agent 语义的监控。

Q035

项目的边界是什么？

参考回答：它监控和展示 agent 状态，不负责控制 agent 执行，也不修改用户代码。这个边界有利于降低安全风险和实现复杂度。

Q036

为什么项目强调“本地读取”？

参考回答：agent transcript、token cache、进程信息本来就在用户机器上。本地读取避免上传敏感代码路径和会话内容，也减少后端维护成本。

Q037

项目如何体现持续迭代？

参考回答：从改安装源、改品牌、扩展 Cursor、调 working 响应、改官网，到 Windows 端 UI 细节优化，形成了真实产品迭代路径。

Q038

如果面试官只给 30 秒，怎么回答？

参考回答：Seeubot 是我做的跨平台 AI Agent 桌面监控组件，支持 Claude、Codex、Cursor，本地读取会话和 token 数据，用 macOS 灵动岛和 Windows 胶囊组件实时展示 working/idle 状态，并提供一行命令安装。

Q039

如果面试官问项目最难的一点，怎么回答？

参考回答：最难的是跨工具和跨平台的一致性：不同 agent 数据源不同，macOS 和 Windows 窗口模型不同，但用户看到的状态和交互要尽量一致。

2. 总体架构与跨平台策略

41 questions
Q040

项目整体架构如何划分？

参考回答：整体分为采集层、状态模型层、UI 展示层和安装分发层。macOS 使用 Swift/SwiftUI 原生实现，Windows 使用 Tauri + React/TypeScript + Rust，并在仓库中通过 apps/windows 与 mac 代码隔离。

Q041

为什么 Windows 端没有直接复用 macOS 代码？

参考回答：macOS 使用 SwiftUI/AppKit，Windows 更适合 Tauri/Rust/Web UI。直接复用会引入复杂跨平台抽象，反而影响稳定性，所以选择同仓库隔离实现，复用概念和数据模型。

Q042

同仓库多端代码有什么好处？

参考回答：可以统一 README、官网、安装入口和版本管理，同时清晰隔离 macOS 与 Windows 源码。对个人项目来说维护成本低，也便于面试官查看完整演进。

Q043

为什么 Windows 放在 apps/windows？

参考回答：apps/windows 明确表达这是独立应用子项目，不和 macOS Sources 混在一起。它让仓库结构更清晰，也为未来新增 apps/linux 或 shared 包留空间。

Q044

跨平台共享了哪些东西？

参考回答：共享的是产品语义、数据结构和交互目标，比如 DashStats、LiveSession、ToolStat、working/idle、token breakdown。代码实现则按平台各自最佳实践完成。

Q045

为什么没有抽一个 shared core？

参考回答：当前 macOS 和 Windows 技术栈差异大，抽 shared core 会增加构建复杂度。项目处于个人产品阶段，先保证端上体验和交付速度更重要。

Q046

未来什么时候适合抽 shared core？

参考回答：当需要支持第三个平台、agent adapter 数量增加、状态模型频繁重复修改时，就适合抽出共享 schema 或 Rust core。当前重复量还可控。

Q047

项目的数据流是怎样的？

参考回答：采集器定期读取本地 transcript、usage cache 和进程信息，生成 DashStats。UI 订阅状态变化，胶囊态展示核心计数，展开态展示 token、工具分布和 live session。

Q048

DashStats 的作用是什么？

参考回答：DashStats 是 UI 和采集层之间的稳定数据契约，包含总 session、working/idle、token、perTool 和 sessions。它让 UI 不需要理解底层文件解析细节。

Q049

LiveSession 的作用是什么？

参考回答：LiveSession 表示一个当前可展示的 agent 会话，包含工具、项目、状态和 token 等信息。它支撑展开态 Live Sessions chips。

Q050

ToolStat 的作用是什么？

参考回答：ToolStat 聚合每个 agent 的会话数量、working/idle 和 token 使用。它用于动态展示 Claude、Codex、Cursor，而不是写死两个工具。

Q051

为什么要从 enum Tool 改成 AgentDescriptor？

参考回答：enum 适合固定工具集合，但扩展 Cursor 后需要动态 agent 描述。AgentDescriptor 用 id、display、glyph 表达 agent，更适合未来增加新工具。

Q052

AgentAdapter 模式解决了什么问题？

参考回答：它把文件枚举、文件解析、进程识别、session 匹配和额外 session 逻辑封装到每个 agent 自己的 adapter 中，避免在主 collect 流程里堆大量 if/else。

Q053

主 collect 流程应该保持什么职责？

参考回答：主流程负责统一调度 adapters、合并 token、构造 sessions 和 perTool。具体某个 agent 怎么识别文件和进程，应交给 adapter。

Q054

如何保证新增 agent 不影响 Claude/Codex 原逻辑？

参考回答：把 Claude/Codex 原有逻辑封装进各自 adapter，并尽量不改它们的解析细节。新增 Cursor 只通过新增 adapter 和 extraSessions 接入。

Q055

为什么 Cursor 使用 extraSessions？

参考回答：Cursor 是 GUI/IDE 场景，主进程不能直接代表每个 agent session。extraSessions 可以从 transcript 生成逻辑会话，而不是依赖进程数量。

Q056

为什么 UI 要动态遍历 perTool？

参考回答：如果 UI 写死 Claude/Codex，新增 Cursor 后仍不会展示。动态遍历 perTool 能让 UI 跟随采集层自然扩展。

Q057

如何决定 agent 是否出现在 perTool？

参考回答：当前策略是有 live session 或有历史 token/记录才显示。这样避免只安装但没使用的工具污染 UI。

Q058

为什么安装但未使用的 agent 不显示？

参考回答：用户关心的是实际使用状态，不是机器上安装了哪些工具。只显示用过或正在运行的 agent 更符合监控产品语义。

Q059

状态刷新周期为什么改成 1 秒？

参考回答：用户反馈 working 状态响应慢，3 秒刷新会让 UI 延迟明显。1 秒刷新能更快反映 agent 开始和结束，同时开销仍可接受。

Q060

刷新过快会有什么风险？

参考回答：过快会增加文件扫描和进程查询开销，Windows 上还可能造成 PowerShell 子进程频繁启动。项目通过隐藏子进程和控制采集范围来缓解。

Q061

为什么 Windows 端采集层用 Rust？

参考回答：Tauri 后端天然使用 Rust，适合做本地文件遍历、JSON 解析和进程调用。Rust 的静态类型也适合表达采集数据结构。

Q062

为什么 Windows UI 用 React/TypeScript？

参考回答：Tauri 的 WebView UI 使用 React/TS 开发效率高，CSS 动画迭代快，适合快速打磨胶囊态、展开态和 bot 动画。

Q063

macOS 为什么用 SwiftUI？

参考回答：原项目是原生 macOS notch widget，SwiftUI/AppKit 能直接控制 NSPanel、menu bar、notch 区域和系统材质。原生实现视觉和窗口行为更自然。

Q064

Windows 为什么没有多模式？

参考回答：Windows 没有 macOS notch 和 menu bar 的同类约束，所以优先做一个顶部居中 Dynamic Island 胶囊，减少模式复杂度。

Q065

为什么 Windows UI 要尽量对齐 macOS？

参考回答：项目是同一个产品，跨端应保持状态语义、视觉反馈和交互节奏一致。否则用户在不同系统上会感到割裂。

Q066

跨端一致性最难的地方是什么？

参考回答：窗口模型不同。macOS 可以用透明 NSPanel 和内部 SwiftUI morph；Windows Tauri 需要处理 WebView、透明窗口、resize 和 hover 事件的异步问题。

Q067

为什么后期把 Windows morph 放到 UI 内部？

参考回答：逐帧调整外层 Windows 窗口尺寸会受系统窗口管理影响，容易卡顿或中间态异常。改成外层窗口即时切尺寸、内部 CSS morph 后响应更稳定。

Q068

为什么外层窗口即时切尺寸不会突兀？

参考回答：用户主要看到的是内部黑色卡片的 morph，而不是透明窗口边界。外层窗口只是承载区域，视觉动画交给内部 UI 完成。

Q069

如何处理 hover 快速进出？

参考回答：使用 phase 状态机和 transition token 取消旧回调，避免旧展开动画在鼠标离开后继续显示内容。后来又加 hoverRef 防止外层窗口扩展后用户已离开仍展开。

Q070

phase 状态机有哪些状态？

参考回答：collapsed、expanding、expanded、collapsing。它把视觉阶段显式化，避免一个 boolean 同时表达多个中间过程。

Q071

为什么不用一个 expanded boolean？

参考回答：boolean 无法区分展开中和已展开、收回中和已收回。中间态不同会影响内容显示、胶囊显示和动画播放。

Q072

transitionToken 有什么作用？

参考回答：每次 hover 状态变化都会递增 token。异步回调执行时检查 token，能防止旧动画回调污染新状态。

Q073

hoverRef 有什么作用？

参考回答：collapsed 时只检测小胶囊或 bar 区域；expanded 后检测整个 panel 区域并加一点 inset。这样避免阅读 dashboard 底部时误收回。

为什么 panel 默认 ignoresMouseEvents？

参考回答：组件大部分区域透明，默认透传能避免挡住用户点击。只有鼠标确实在 widget 内时才接收事件。

NotchMetrics 的作用是什么？

参考回答：它测量屏幕是否有 notch、notchWidth、notchHeight、menuBarHeight 和 screenWidth，从而适配有刘海和无刘海 Mac。

Dim 常量的作用是什么？

Q175

NotchRootView 的核心设计是什么？

Q176

为什么 mac 端动画更丝滑？

参考回答：mac 端外层 panel 尺寸固定，主要在 SwiftUI 内部做 morph。内部动画由 SwiftUI spring 驱动，不依赖系统窗口逐帧 resize。

参考回答：Windows 逐帧 resize 外层窗口容易卡顿。mac 的经验说明应该把视觉 morph 放在内部 UI 层，外层窗口只作为承载。
